Build Samvera stack step by step

Environments

Ubuntu: 18.04.1 LTS
Java: OpenJDK version 11
Tomcat: 9.0.12
Solr: 7.5.0
Fedora: 4.5.1
Ruby: 2.5.3
Rails: 5.2.1

Install OpenJDK version 11

sudo apt install default-jdk

sudo nano /etc/environment

Add: 

JAVA_HOME="/usr/lib/jvm/java-11-openjdk-amd64"

Install Tomcat 9.0.12 as a service

This part is mainly following this post

sudo groupadd tomcat
sudo useradd -s /bin/false -g tomcat -d /opt/tomcat tomcat
cd /tmp
curl -O http://mirror.ox.ac.uk/sites/rsync.apache.org/tomcat/tomcat-9/v9.0.12/bin/apache-tomcat-9.0.12-deployer.tar.gz
sudo mkdir /opt/tomcat
sudo tar xzvf apache-tomcat-9*tar.gz -C /opt/tomcat --strip-components=1
cd /opt/tomcat
sudo chgrp -R tomcat /opt/tomcat
sudo chmod -R g+r conf
sudo chmod g+x conf
sudo chown -R tomcat webapps/ work/ temp/ logs/

Then create a Tomcat service:

sudo nano /etc/systemd/system/tomcat.service

With following content:

[Unit]
Description=Apache Tomcat Web Application Container
After=network.target

[Service]
Type=forking

Environment=JAVA_HOME=/usr/lib/jvm/java-11-openjdk-amd64
Environment=CATALINA_PID=/opt/tomcat/temp/tomcat.pid
Environment=CATALINA_HOME=/opt/tomcat
Environment=CATALINA_BASE=/opt/tomcat
Environment='CATALINA_OPTS=-Xms512M -Xmx1024M -server -XX:+UseParallelGC'
Environment='JAVA_OPTS=-Djava.awt.headless=true -Djava.security.egd=file:/dev/./urandom'

ExecStart=/opt/tomcat/bin/startup.sh
ExecStop=/opt/tomcat/bin/shutdown.sh

User=tomcat
Group=tomcat
UMask=0007
RestartSec=10
Restart=always

[Install]
WantedBy=multi-user.target

Reload system daemon:

sudo systemctl daemon-reload

Start Tomcat:

sudo systemctl start tomcat

Test if you can access Tomcat via port 8080, then enable this service:

sudo systemctl enable tomcat

Configure Tomcat web admin UI:

sudo nano /opt/tomcat/conf/tomcat-users.xml

With the content:

<?xml version="1.0" encoding="UTF-8"?>
<tomcat-users xmlns="http://tomcat.apache.org/xml"
              xmlns:xsi="http://www.w3.org/2001/XMLSchema-instance"
              xsi:schemaLocation="http://tomcat.apache.org/xml tomcat-users.xsd"
              version="1.0">
  <role rolename="manager-gui"/>
  <user username="tomcat" password="PASSWORD" roles="manager-gui"/>

  <role rolename="admin-gui"/>
  <user username="admin" password="PASSWORD" roles="admin-gui"/>
</tomcat-users>

By default, Tomcat adds IP restrictions for Manager UI, either comment it out or add your IP into the list.

sudo nano /opt/tomcat/webapps/manager/META-INF/context.xml

And restart Tomcat

Add additional downloader metadata

module Oasis

    included do
      property :time, predicate: ::RDF::URI.new('https://oasis-database.org/ontologies/oasis#downloader_time'), multiple: false do |index|
        index.as :stored_searchable
      end

      # ......
  end
end

Also, edit app/models/downloader.rb to add:

include ::Oasis::DownloaderMetadata

Setup relationships

Define the relationship between Summary and Downloader:

In Summary,

class Summary < ActiveFedora::Base

In Downloader,

class Downloader < ActiveFedora::Base
  belongs_to :summary, predicate: ::RDF::URI.new('https://oasis-database.org/ontologies/oasis#downloader')

end

How to use blacklight hierarchy gem to implement hierarchical facets

Blacklight hierarchy provides a light way to support hierarchical facets,
but the documentation is a bit hard to understand, I just write down what I have done to get a working prototype.

Update Gemfile

and run:

Define facets in catalog_controller

Edit app/controllers/catalog_controller.rb:

config.add_facet_field solr_name('summary_general_research_area', :facetable),
                       label: "Topic",
                       :partial => 'blacklight/hierarchy/facet_hierarchy'

config.facet_display = {
    :hierarchy => {
        'summary_general_research_area' => [['sim'],':']
    }
}

NB: the facet field name in solr is concatenated from above ‘summary_general_research_area’ and ‘sim’,
e.g. the solr facet field name is: ‘facet-summary_general_research_area_sim’

Adding a new static page in Hyrax

Hyrax provides a nice feature to let users define the content of static pages as content blocks. But sometimes, users may want to add a new page and keep updating the content of that page by non-technical people. Here is how:


Image source: pixabay.com

Register the new page in ContentBlock

Edit file: app/models/content_block.rb

class ContentBlock < ActiveRecord::Base
  NAME_REGISTRY = {
    #......
    activities: :activities_page
  }.freeze

  class << self
    #......
    def activities_page
       find_by(name: 'activities_page') ||
          create(name: 'activities_page', value: default_activities_text)
    end

    def activities_page=(value)
       activities_page.update(value: value)
    end

    #......
    def default_activities_text
      ERB.new(
          IO.read(
              Rails.root.join('app', 'views', 'hyrax', 'content_blocks', 'templates', 'activities.html.erb')
          )
      ).result
    end
  end  
end

Register the new page in page controller

Copy app/controllers/hyrax/pages_controller.rb from Hyrax to your APP, and add the following information:

module Hyrax
  class PagesController < ApplicationController
    # ......

    private
      def permitted_params
          params.require(:content_block).permit(:about,
                                                :agreement,
                                                :help,
                                                :terms,
                                                :activities)
      end
  end
end

Add a page for default content

Create a new file and put any content you like: app/views/hyrax/content_blocks/templates/activities.html.erb

Make changes to UI

Copy file app/views/hyrax/pages/_form.html.erb from Hyrax to your APP and edit:

<div class="panel panel-default tabs">
  <ul class="nav nav-tabs" role="tablist">
    <!-- ...... -->
    <li>
      <a href="#activities" role="tab" data-toggle="tab"><%= t(:'hyrax.pages.tabs.activities_page') %></a>
    </li>
  </ul>

  <div class="tab-content">
      <!--......-->
      <div id="activities" class="tab-pane">
        <div class="panel panel-default labels">
          <%= simple_form_for ContentBlock.for(:activities), url: hyrax.page_path(ContentBlock.for(:activities)) do |f| %>
              <div class="panel-body">
                <div class="field form-group">
                  <%= f.label :activities %><br>
                  <%= f.text_area :activities, value: f.object.value, class: 'form-control tinymce', rows: 20, cols: 120 %>
                </div>
              </div>
              <div class="panel-footer">
                <%= link_to t(:'hyrax.pages.cancel'), hyrax.admin_admin_sets_path, class: 'btn btn-default pull-right'%>
                <%= f.button :submit, class: 'btn btn-primary pull-right'%>
              </div>
          <% end %>
        </div>
      </div>
  </div>

Define labels for the new page

Edit config/locales/hyrax.en.yml

en:
  hyrax:
    pages:
      tabs:
        activities_page: "OASIS activities and presentations"
        people_saying_page: "What people say about OASIS"

Update routes

Edit config/routes.rb

get 'activites' => 'hyrax/pages#show', key: 'activites'
get 'people_saying' => 'hyrax/pages#show', key: 'people_saying'

Define customized Samvera models in OASIS, step by step

I’ve been using Samvera stack since I’ve started working on OASIS. Building a customized model is a complex job, so I’d write down what I’ve learned, in case it might be useful for others.

Create model in Ruby

It’s a good idea to define model attributes in a seperate file, e.g. 

module Oasis
  module OasisMetadata extend ActiveSupport::Concern
    included do
      property :summary_writer, predicate: ::RDF::URI.new('XXXXXX'), multiple: true do |index|
        index.as :stored_searchable, :sortable, :facetable
      end

      # OTHER ATTRIBUTES ...
    end
  end    
end

Source code is here

And include it in the main model:

class Summary < ActiveFedora::Base
  include ::Hyrax::WorkBehavior
  include ::Oasis::OasisMetadata
end

Source code is here

Define SolrDocument

The attributes should be exposed in SolrDocument. By this way, these attributes are accessable directly from SolrDocument in the view later.

module Oasis
  module SolrDocument extend ActiveSupport::Concern
  module Solr
    class Array
      # @return [Array]
      def self.coerce(input)
        ::Array.wrap(input)
      end
    end

    class String
      # @return [String]
      def self.coerce(input)
        ::Array.wrap(input).first
      end
    end

    class Date
      # @return [Date]
      def self.coerce(input)
        field = String.coerce(input)
        return if field.blank?
        begin
          ::Date.parse(field)
        rescue ArgumentError
          Rails.logger.info "Unable to parse date: #{field.first.inspect}"
        end
      end
    end
  end

  included do
    attribute :title_of_summary, Solr::String, solr_name('title_of_summary')
    attribute :summary_writer, Solr::Array, solr_name('summary_writer')
    attribute :summary_writer_all, Solr::String, solr_name('summary_writer_all')
    # ......
  end  
  end
end

Source code

To access these attributes in the view:

<h4><%= link_to document.title_of_summary, document %></h4>

Define customized Solr Indexer

Customized Solr Indexer can be defined to store extra information into solr, e.g. generate a new field in Solr, add a derived value into Solr etc.

class SummaryIndexer < Hyrax::WorkIndexer
  include Hyrax::IndexesBasicMetadata
  include Hyrax::IndexesLinkedMetadata

  # add custom indexing behavior for authority fields:
  def generate_solr_document
    super.tap do |solr_doc| 
      #DO your stuff here.....
    end
  end    
end

The customized Indexer needs to be registered in your model, e.g.

class Summary < ActiveFedora::Base
  include ::Hyrax::WorkBehavior

  self.indexer = SummaryIndexer
  #......
end

    # Add OASIS authority label fields to Solr query parameters
    def add_authority_label_field_to_query(solr_parameters)
      #......
    end

    # Add OASIS text fields to Solr query parameters
    def add_oasis_text_fields_to_query(solr_parameters)
      solr_parameters[:qf] += ' publication_author_tesim'
      solr_parameters[:qf] += ' publication_title_tesim'

      solr_parameters[:qf] += ' title_of_summary_tesim'
    end

    end
